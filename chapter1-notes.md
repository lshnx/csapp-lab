# CS:APP 第一章笔记 — 计算机系统漫游

## 1.1 信息就是位 + 上下文

- 只由 ASCII 字符构成的文件称为 **文本文件**，所有其他文件称为 **二进制文件**
- 系统中的所有信息——磁盘文件、存储器中的程序、存储器中存放的用户数据以及网络上传送的数据——都是由一串比特（bit）表示的
- 区分不同数据对象的唯一方法是读到这些数据对象的**上下文**（context）

> 理解：同样的 0101，在不同上下文中含义完全不同。比如 4 个字节的比特串，可能是整数、浮点数、一条指令、一个地址……关键在于程序怎么解释它。

## 1.2 程序被其他程序翻译成不同的格式

hello.c 变成可执行文件的过程（四个阶段）：

| 阶段 | 工具 | 输入 | 输出 | 做了什么 |
|------|------|------|------|----------|
| 1. 预处理 | 预处理器（cpp） | hello.c | hello.i | 把 `#include` `#define` 等 # 开头的命令展开 |
| 2. 编译 | 编译器（cc1） | hello.i | hello.s | 把 C 代码翻译成汇编语言 |
| 3. 汇编 | 汇编器（as） | hello.s | hello.o | 把汇编翻译成机器指令，打包成可重定位目标文件（二进制） |
| 4. 链接 | 链接器（ld） | hello.o + 其他 .o | hello | 把多个目标文件合并成一个可执行文件 |

> 简化记忆：**预 → 编 → 汇 → 链**

## 1.3 了解编译系统如何工作是大有益处的

- **优化程序性能**：C 代码和机器指令不是一一对应的。理解编译器怎么翻译你的代码，才知道怎么写快代码
- **理解链接时出现的错误**：`undefined reference` 这类经典报错，根源在链接阶段。不懂链接原理就永远在靠猜改 bug
- **避免安全漏洞**：栈溢出攻击本质上是利用程序在内存中的表示缺陷。不懂底层就写不出安全的代码

## 1.4 处理器读并解释储存在存储器中的指令

### shell 是什么
shell 是一种命令行解释器，它输出一个提示符，等待你输入一行命令，然后执行。如果该命令行的第一个单词不是内置的 shell 命令，shell 就假设这是一个可执行文件的名字，加载并执行它。

## 1.4.1 系统的硬件组成

### 总线（Bus）
贯穿整个系统的一组电子管道，负责在各个部件之间传递信息字节。总线被设计成传送定长的字节块，也就是 **"字"（word）**。一个字中的字节数（字长）是系统的基本参数，不同系统字长不同（如 4 字节 / 8 字节）。

### I/O 设备
系统与外界的联系通道：
- 输入设备：键盘、鼠标
- 输出设备：显示器
- 存储设备：磁盘驱动器（用于长期存储数据和程序）

每个 I/O 设备通过**控制器**或**适配器**与 I/O 总线连接：
- 控制器：I/O 设备本身中或主板上的芯片组
- 适配器：插在主板插槽上的卡

> 二者功能一样：在 I/O 总线和 I/O 设备之间传递信息。区别只是封装方式不同。

### 主存（Main Memory）
处理器执行程序时用来临时存放程序和处理的数据的设备。

- 物理上：由一组 **DRAM**（动态随机存取存储器）芯片组成
- 逻辑上：一个线性的字节数组，每个字节有唯一的地址（从 0 开始），地址就是数组索引
- 组成程序的每条机器指令由不定数量的字节构成
- C 中的 `int` 类型数据在大多数机器上占 4 个字节

### 处理器（CPU）
中央处理单元，是**解释（或执行）存储在主存中指令**的引擎。

三个核心部件：
- **PC（程序计数器）**：一个字长大小的寄存器，任何时候都指向主存中的某条机器语言指令（即存放该指令的地址）
- **寄存器文件**：一个小型存储设备，由若干字长大小的寄存器组成，每个寄存器有唯一的名字
- **ALU（算术逻辑单元）**：执行算术和逻辑运算，计算新的数据和地址值

CPU 的基本工作循环（一条指令的执行过程）：
1. 从 PC 指向的内存位置读取指令
2. 解释指令中的位（判断这条指令要干什么）
3. 执行该指令指示的简单操作（加载、存储、运算、跳转等）
4. 更新 PC，指向下一条指令（下一条不一定和当前指令相邻——遇到跳转指令时 PC 会跳到别处）

## 1.4.2 运行 hello 程序

### 完整的执行流程

从键盘敲下 `./hello` 到屏幕上出现 `hello, world`，分两大步：

### 第一步：Shell 把 hello 文件从磁盘加载到主存

1. 键盘输入 `./hello` 回车 → 字符经 USB 控制器、I/O 总线进入 CPU 寄存器 → 最终存入主存
2. Shell 判断 `./hello` 不是内置命令 → 认为它是可执行文件
3. Shell 通过 **DMA（直接存储器访问）** 把 hello 可执行文件的代码和数据从磁盘搬到主存
   - DMA 的含义：数据从磁盘直接到主存，**不经过 CPU**
   - CPU 只需要发一条命令告诉磁盘控制器"把文件读到地址 X"，然后去做别的事
   - 磁盘控制器自己把数据搬完，完成后通知 CPU

> 如果没有 DMA，CPU 就得一个字节一个字节地从磁盘读到寄存器再写到主存，整个过程中 CPU 卡死干不了别的。

### 第二步：CPU 逐条执行 hello 的机器指令

hello 加载到主存后，CPU 开始工作：

| 步骤 | 做什么 | 谁干的 |
|------|--------|--------|
| 取指（Fetch） | 从 PC 指向的主存位置读一条指令 | CPU 控制单元 |
| 译码（Decode） | 判断这条指令要干什么（mov? add? syscall?） | CPU 控制单元 |
| 执行（Execute） | 在 ALU 中做运算，或在寄存器与主存之间搬数据 | ALU / 寄存器 |
| 更新 PC | PC 指向下一条指令（顺序执行 or 跳转） | CPU |

hello 程序本质上是若干条 `mov` + 一条 `syscall`：
- 把 `1`（write 系统调用号）放进寄存器 → 把 `1`（stdout）放进寄存器 → 把字符串地址放进寄存器 → 把长度 `13` 放进寄存器
- 最后一条 `syscall`：让操作系统代劳，把 13 个字节写到显示器

### 各硬件组件的角色回顾

| 组件 | 在 hello 执行中干了什么 |
|------|------------------------|
| 总线 | 键盘信号走 I/O 总线；DMA 走总线搬文件；结果写显示器也走总线 |
| I/O 设备 | 键盘（输入命令）、磁盘（存储 hello 文件）、显示器（输出结果） |
| 主存 | 存放 shell 代码 + hello 代码 + hello 数据（字符串 "hello, world\n"） |
| CPU | PC 指向当前指令、寄存器暂存数据、ALU 做运算、逐条执行 |
| DMA | 磁盘 → 主存搬运不占 CPU，让 CPU 同时干别的事 |

> 一句话总结：敲下 `./hello` → Shell 通过 DMA 把可执行文件从磁盘搬到主存 → CPU 逐条取指/译码/执行 → 最终通过系统调用把 "hello, world\n" 写到显示器。

## 1.5 高速缓存至关重要

### 问题：CPU 太快，主存太慢

| | 速度 | 差距 |
|---|---|---|
| CPU 执行一条指令 | ~1 ns | — |
| 从主存读一个数据 | ~100 ns | 比 CPU 慢 100 倍 |
| 从磁盘读一个数据 | ~10,000,000 ns | 比主存慢 10 万倍 |

> CPU 每次去主存取数据，就像你跑得飞快但每次都要等一个慢吞吞的电梯送东西过来。CPU 大部分时间在等待，而不是在干活。

### 解决方案：在 CPU 和主存之间加一层小但快的中转站

这个中转站就是 **高速缓存（cache）**。

```
CPU  ←→  L1 cache  ←→  L2 cache  ←→  L3 cache  ←→  主存  ←→  磁盘
       (最快/最小)   (次快/次小)   (再次)          (慢/大)    (最慢/最大)
```

| 层级 | 大小 | 访问时间 | 位置 |
|------|------|----------|------|
| L1 cache | 几十 KB | ~1 ns | CPU 芯片内部 |
| L2 cache | 几百 KB | ~5 ns | CPU 芯片内部 |
| L3 cache | 几 MB | ~20 ns | CPU 芯片内部 |
| 主存 | 几 GB ~ 几十 GB | ~100 ns | 主板上 |
| 磁盘 | 几百 GB ~ TB | ~10 ms | 机箱里 |

### Cache 凭什么能加速？—— 局部性原理（locality）

程序访问数据不是随机的，有两个规律：

1. **时间局部性**：刚用过的数据，短时间内很可能再用
   - 比如循环里的变量 `i`，每轮都要访问
2. **空间局部性**：用了地址 X 的数据，地址 X+1 的数据也快被用了
   - 比如数组 `arr[0] arr[1] arr[2]...`，连续存放，连续访问

Cache 利用这两个规律：把最近用过的数据和它附近的数据都缓存起来，后续访问大概率命中。

> 理解：Cache 就像一个聪明的小助手——你刚看了一本书的第 10 页，它不仅帮你把第 10 页摊在桌上（时间局部性），还顺手把第 11、12 页也翻开备好（空间局部性）。下次你要第 11 页的时候，不用再去书架上找。

### 为什么这对你重要

写高性能代码的关键就是**让数据尽量待在 cache 里**：
- 遍历二维数组要按行遍历（内存连续），不要按列遍历（跳着访问，cache miss 多）
- 同一个数据反复用就别反复从主存取

> 对 AI Infra 的意义：GPU 上写 CUDA kernel，核心优化就是把数据从 global memory（慢，类比主存）搬到 shared memory（快，类比 L1 cache），让计算单元吃饱数据不等饭。

## 1.6 存储设备形成层次结构

### 整个计算机的存储体系

从上到下，越来越慢、越来越大、越来越便宜：

```
  ┌──────────────┐
  │   寄存器      │  ← CPU 内部，几百字节，0 延迟
  ├──────────────┤
  │  L1 cache    │  ← SRAM，CPU 芯片上
  ├──────────────┤
  │  L2 cache    │  ← SRAM
  ├──────────────┤
  │  L3 cache    │  ← SRAM
  ├──────────────┤
  │  主存 (DRAM)  │  ← 插在主板上
  ├──────────────┤
  │ 本地磁盘(SSD) │  ← 机箱里
  ├──────────────┤
  │ 远程存储     │  ← 网络上的分布式文件系统
  └──────────────┘

  规律：每一层都是上一层的"cache"
  - 主存 是 磁盘 的 cache
  - L3 是 主存 的 cache
  - L2 是 L3 的 cache
  - 以此类推
```

> 核心思想：程序员可以把整个存储系统看成一个**按层次组织的 cache 金字塔**。写程序时利用好这个层次结构，数据尽量待在金字塔上层，性能就好。

## 1.7 操作系统管理硬件

### 操作系统是硬件和应用程序之间的中间层

```
┌──────────────────────────┐
│      应用程序 (hello)     │
├──────────────────────────┤
│      操作系统 (OS)        │  ← 抽象层，屏蔽硬件细节
├──────────────────────────┤
│   CPU / 内存 / I/O 设备   │
└──────────────────────────┘
```

OS 提供三大抽象：

| 抽象 | 对应什么 | 解决了什么问题 |
|------|----------|---------------|
| **进程** | CPU + 主存 + I/O 设备 | 让你以为程序独占 CPU |
| **虚拟内存** | 主存 + 磁盘 | 让你以为程序独占全部内存 |
| **文件** | I/O 设备 | 让所有设备看起来都像读写文件 |

### 进程

进程是**操作系统对一个正在运行的程序的抽象**。

- 一个系统可以同时运行多个进程（你开着浏览器 + VS Code + 终端）
- 每个进程都以为自己独占 CPU——这是 OS 用**上下文切换**制造的假象
- OS 保存当前进程的状态（PC、寄存器、内存内容），切换到另一个进程，再切回来

```
进程 A ████░░░░░░░░████░░░░░░░░████
进程 B ░░░░████████░░░░████████░░░░
        ← 时间片轮转，宏观上"同时"跑 →
```

> 关键：上下文切换是由 OS 内核管理的，应用程序感受不到切换。

### 线程

一个进程可以由多个**线程**组成：
- 每个线程有自己独立的 PC 和寄存器（执行不同的代码流）
- 同一进程的线程**共享虚拟内存**（可以访问同一份数据）
- 线程之间切换比进程切换更快

> 对 AI Infra 的意义：PyTorch 训练用多线程加载数据（DataLoader 的 `num_workers`），CUDA 编程里一个 kernel 跑几千个线程——线程是现代并发的核心单元。

### 虚拟内存

每个进程看到的内存地址都是**假的**——是 OS 给它的虚拟地址空间。

```
进程 A 看到的：              进程 B 看到的：
┌──────────────┐            ┌──────────────┐
│ 0x0000...    │            │ 0x0000...    │
│   代码       │            │   代码       │
│   数据       │            │   数据       │
│   堆         │            │   堆         │
│   栈         │            │   栈         │
│ 0xFFF...     │            │ 0xFFF...     │
└──────────────┘            └──────────────┘
    (虚拟的)                    (虚拟的)
         ↘                    ↙
          ┌──────────────┐
          │  物理主存     │  ← 实际的内存条
          │  + 磁盘交换    │
          └──────────────┘
```

> 虚拟内存在 AI Infra 里极其重要：GPU 显存管理、CUDA 的 unified memory、NUMA 架构都依赖虚拟内存机制。

### 文件

文件是对 I/O 设备的统一抽象——**一切皆文件**。

- 磁盘上的普通文件是文件
- 键盘、显示器、网络接口也是"文件"
- 程序用 `open / read / write / close` 这套统一接口操作所有设备

> 这就是 Unix 哲学的体现：不管你连的是键盘还是网卡，程序眼里都是"一个文件"，读写就行。

## 第一章总结

一个 hello 程序的完整旅程：

```
hello.c  →  预处理  →  编译  →  汇编  →  链接  →  hello (可执行文件)
                                                      ↓
                                               shell 加载到主存 (DMA)
                                                      ↓
                                               CPU 逐条执行指令
                                               ↕ (cache 加速)
                                             寄存器 ← L1 ← L2 ← L3 ← 主存
                                                      ↓
                                              系统调用 → 写显示器
```

**第一章的核心画面**：计算机 = 硬件（总线 + CPU + 主存 + I/O）在上面跑、操作系统在中间管、程序在最上面。你要做的就是理解每一层在干什么，以及它们怎么协作。
