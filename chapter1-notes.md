## 1.1 信息就是位 + 上下文

- 只由 ASCII 字符构成的文件称为**文本文件**，所有其他文件称为**二进制文件**
- 系统中的所有信息——磁盘文件、存储器中的程序、存储器中存放的用户数据及网络上传送的数据——都由一串**比特（bit）** 表示
- 区分不同数据对象的唯一方法是读到这些数据对象时的**上下文（context）**

> **理解**：同样的 0101，在不同上下文中含义完全不同。4 个字节的比特串，可能是整数、浮点数、一条指令、一个地址……关键在于程序怎么解释它。这是贯穿全书的核心理念。

## 1.2 程序被其他程序翻译成不同的格式

hello.c 变成可执行文件的过程（GCC 驱动四个阶段）：

| 阶段 | 工具 | 输入 | 输出 | 做了什么 |
|------|------|------|------|----------|
| **预处理** | 预处理器（cpp） | hello.c | hello.i | 展开 `#include`、`#define` 等 `#` 开头的命令，插入头文件内容 |
| **编译** | 编译器（cc1） | hello.i | hello.s | 把 C 代码翻译成汇编语言（文本） |
| **汇编** | 汇编器（as） | hello.s | hello.o | 把汇编翻译成机器指令，打包成**可重定位目标文件**（二进制） |
| **链接** | 链接器（ld） | hello.o + libc.o | hello（可执行） | 把多个 .o 文件合并，解析函数调用地址（如 `printf`） |

> 简化记忆：**预 → 编 → 汇 → 链**。GCC 一条 `gcc hello.c` 背后是这四个阶段的流水线。

GCC 常用分步选项：
```bash
gcc -E hello.c -o hello.i   # 只预处理
gcc -S hello.i -o hello.s    # 只编译（生成汇编）
gcc -c hello.s -o hello.o    # 只汇编（生成目标文件）
gcc hello.o -o hello          # 只链接
```

## 1.3 了解编译系统如何工作是大有益处的

书中给出三个理由：

1. **优化程序性能**：C 代码和机器指令不是一一对应的。理解编译器如何翻译代码，才知道怎么写快代码（如循环展开、减少内存引用）。
2. **理解链接时错误**：`undefined reference to 'foo'`、`multiple definition of 'bar'`——根源在链接阶段。不懂链接原理就只能靠猜修 bug。
3. **避免安全漏洞**：栈溢出攻击利用的是程序在内存中的表示缺陷（第 3 章详讲）。不懂底层就写不出安全的代码。

## 1.4 处理器读并解释储存在存储器中的指令

### shell 是什么

shell 是命令行解释器——输出提示符，等待输入，执行命令。如果命令的第一个单词不是内置命令，shell 就假设它是可执行文件名，加载并运行它。

---

### 1.4.1 系统的硬件组成

#### 总线（Bus）

贯穿整个系统的一组电子管道，在部件间传递信息。总线传送定长的字节块——**字（word）**。字长（word size）是系统基本参数：32 位系统 4 字节，64 位系统 8 字节。

#### I/O 设备

系统与外界的通道：
- 输入：键盘、鼠标
- 输出：显示器
- 存储：磁盘驱动器

每个 I/O 设备通过**控制器**或**适配器**与 I/O 总线连接：
- **控制器**：I/O 设备本身或主板上的芯片组
- **适配器**：插在主板插槽上的卡

> 二者功能相同：在 I/O 总线和设备间搬运数据。区别只在封装形式。

#### 主存（Main Memory）

处理器执行程序时临时存放程序和数据的地方。

- 物理上：一组 **DRAM** 芯片
- 逻辑上：线性的字节数组，每个字节有唯一地址（从 0 开始）
- 一条机器指令由不定数量的字节组成
- C 的 `int` 在大多数机器上占 4 字节

#### 处理器（CPU）

中央处理单元 = 解释/执行主存中指令的引擎。

三个核心部件：
| 部件 | 作用 |
|------|------|
| **PC（程序计数器）** | 字长大小的寄存器，始终指向主存中下一条要执行的指令的地址 |
| **寄存器文件（Register File）** | 小型存储设备，由若干字长大小的寄存器组成，每个有唯一名字 |
| **ALU（算术逻辑单元）** | 执行算术和逻辑运算，计算新数据和地址值 |

CPU 的基本工作循环：
1. 从 PC 指向的内存位置**读取**指令
2. **解释**指令中的位（判断做什么操作）
3. **执行**该指令指示的简单操作（加载/存储/运算/跳转）
4. **更新** PC 指向下一条指令（下一条不一定相邻——跳转指令会改 PC）

> CPU 看起来执行了很多复杂操作，但底层只有这几类简单指令。第 3、4 章会深入整个指令集体系结构（ISA）。

### 1.4.2 运行 hello 程序

从敲下 `./hello` 到屏幕出现 `hello, world`，分两大步：

#### 第一步：Shell 把 hello 从磁盘加载到主存

1. 键盘输入 `./hello` → 字符经 USB 控制器 → I/O 总线 → CPU 寄存器 → 主存
2. Shell 判断 `./hello` 不是内置命令 → 视其为可执行文件
3. Shell 通过 **DMA（直接存储器访问）** 把 hello 的代码和数据从磁盘搬到主存
   - DMA：数据从磁盘直接到主存，**不经过 CPU**
   - CPU 只需发一条命令告诉磁盘控制器「读到地址 X」，然后干别的
   - 磁盘控制器搬完数据后通过中断通知 CPU

> 没有 DMA 的话，CPU 得一字节一字节从磁盘读到寄存器再写进主存，全程卡死干不了别的。

#### 第二步：CPU 逐条执行 hello 的机器指令

| 步骤 | 做什么 | 硬件 |
|------|--------|------|
| 取指（Fetch） | 从 PC 指向的主存位置读一条指令 | 控制单元 |
| 译码（Decode） | 判断这条指令要干什么（mov? add? syscall?） | 控制单元 |
| 执行（Execute） | ALU 做运算 / 寄存器与主存间搬数据 | ALU + 寄存器 |
| 更新 PC | PC 指向下一条指令（顺序 or 跳转） | 控制单元 |

hello 程序的本质：若干条 `mov`（把系统调用号和参数装进寄存器）+ 一条 `syscall`（让操作系统代劳写显示器）。

#### 各硬件组件角色回顾

| 组件 | 在 hello 执行中的角色 |
|------|----------------------|
| 总线 | 键盘信号走 I/O 总线；DMA 走总线搬文件；写显示器也走总线 |
| I/O 设备 | 键盘（输入）、磁盘（存储 hello）、显示器（输出） |
| 主存 | 存放 shell 代码 + hello 代码 + 字符串 `"hello, world\n"` |
| CPU | PC 指向当前指令、寄存器暂存数据、ALU 运算、逐条执行 |
| DMA | 磁盘→主存搬运不占 CPU，让 CPU 同时做别的事 |

> 一句话总结：敲 `./hello` → Shell 通过 DMA 把可执行文件从磁盘搬到主存 → CPU 逐条取指/译码/执行 → 系统调用把 `"hello, world\n"` 写到显示器。

## 1.5 高速缓存至关重要

### 问题：CPU 太快，主存太慢

| | 大致延迟 | 相对 CPU 周期 |
|---|---|---|
| CPU 执行一条指令 | ~1 ns | 1× |
| 从主存读一个数据 | ~100 ns | ~100× |
| 从磁盘读一个数据 | ~10,000,000 ns | ~10,000,000× |

> CPU 大部分时间在**等数据**，不是在算数据。这就是「存储器墙」（Memory Wall）。

### 解决方案：高速缓存（Cache）

在 CPU 和主存之间插入更小、更快的存储设备——SRAM 构成的 cache。

```
CPU  ←→  L1 cache  ←→  L2 cache  ←→  L3 cache  ←→  主存  ←→  磁盘
      (最快/最小)    (次快/次小)    (再次/几MB)      (慢/GB)    (最慢/TB)
```

| 层级 | 大小 | 访问时间 | 位置 |
|------|------|----------|------|
| L1 cache | 几十 KB | ~1 ns（~4 cycles） | CPU 芯片内部 |
| L2 cache | 几百 KB | ~5 ns（~10 cycles） | CPU 芯片内部 |
| L3 cache | 几 MB | ~20 ns（~40 cycles） | CPU 芯片内部（多核共享） |
| 主存 | 几 GB ~ 几十 GB | ~100 ns（~200 cycles） | 主板 DIMM 插槽 |
| 磁盘（SSD） | 几百 GB ~ TB | ~10-100 μs | 机箱里 |

### Cache 为什么有效？——局部性原理（Locality）

程序访问数据不是随机的，有两个规律：

1. **时间局部性（Temporal Locality）**：刚用过的数据，短时间内很可能再用。如循环变量 `i`。
2. **空间局部性（Spatial Locality）**：用了地址 X 的数据，地址 X+1 的数据很快也会被用。如数组连续访问。

> Cache 就是一个聪明的小助手——你刚看了书的第 10 页，它不仅把第 10 页摊在桌上（时间局部性），还顺手把第 11、12 页翻开备好（空间局部性）。

### 为什么这对你重要

写高性能代码的核心 = **让数据尽量待在 cache 里**：
- 遍历二维数组按行（内存连续），不要按列（跳着访问 → cache miss 多）
- 同一数据反复用就别反复从主存取（编译器用寄存器优化帮你做了一部分）

> **对 AI Infra 的意义**：GPU 上写 CUDA kernel，核心优化就是把数据从 global memory（慢，~几百 cycles）搬到 shared memory（快，L1 cache 级别），让计算单元吃饱数据不等饭。

## 1.6 存储设备形成层次结构

从上到下：越来越慢、越来越大、越来越便宜。

```
  ┌──────────────┐
  │   寄存器      │  ← CPU 内部，几百字节，0 延迟
  ├──────────────┤
  │  L1 cache    │  ← SRAM，CPU 芯片上
  ├──────────────┤
  │  L2 cache    │  ← SRAM
  ├──────────────┤
  │  L3 cache    │  ← SRAM
  ├──────────────┤
  │  主存 (DRAM)  │  ← 主板 DIMM
  ├──────────────┤
  │ 本地 SSD/HDD  │  ← 机箱里
  ├──────────────┤
  │ 远程存储     │  ← 网络分布式文件系统
  └──────────────┘

  核心规律：每一层都是上一层的"cache"
  - 主存是磁盘的 cache
  - L3 是主存的 cache
  - L2 是 L3 的 cache
  - 寄存器是 L1 的 cache
```

> 程序员可以把整个存储体系看成一个**按层次组织的 cache 金字塔**。利用好这个层次——让数据尽量待在金字塔上层——性能就好。

## 1.7 操作系统管理硬件

### 操作系统是硬件和应用程序之间的中间层

```
┌──────────────────────────┐
│      应用程序 (hello)     │
├──────────────────────────┤
│      操作系统 (OS)        │  ← 抽象层，屏蔽硬件细节
├──────────────────────────┤
│   CPU / 内存 / I/O 设备   │
└──────────────────────────┘
```

OS 提供三大抽象（贯穿全书的核心概念）：

| 抽象 | 对应硬件 | 解决的问题 |
|------|----------|-----------|
| **进程** | CPU + 主存 + I/O 设备 | 让你以为程序独占 CPU |
| **虚拟内存** | 主存 + 磁盘 | 让你以为程序独占全部内存 |
| **文件** | I/O 设备 | 让所有设备看起来都像读写文件 |

---

### 进程

进程是**操作系统对一个正在运行的程序的抽象**。

- 一个系统可以同时运行多个进程（浏览器 + VS Code + 终端）
- 每个进程都以为自己独占 CPU——这是 OS 用**上下文切换（context switch）** 制造的假象
- OS 保存当前进程的状态（PC、寄存器、内存页表），切换到另一个进程，再切回来

```
进程 A ████░░░░░░░░████░░░░░░░░████
进程 B ░░░░████████░░░░████████░░░░
        ← 时间片轮转，宏观上"同时"跑 →
```

> 关键：上下文切换由 OS **内核（kernel）** 管理，应用程序感受不到。

### 线程

一个进程可由多个**线程**组成：
- 每个线程有独立的 PC 和寄存器（可执行不同代码流）
- 同一进程的线程**共享虚拟地址空间**（访问同一份数据）
- 线程切换比进程切换更轻量（不需要切换页表）

> **对 AI Infra 的意义**：PyTorch DataLoader 的 `num_workers` 用多进程（不是多线程，因为 Python GIL）加载数据；CUDA 里一个 kernel 跑几千个线程——线程是现代并发的核心单元。

### 虚拟内存

每个进程看到的内存地址都是**虚拟地址**——OS 给的假地址空间。

```
进程 A 看到的：              进程 B 看到的：
┌──────────────┐            ┌──────────────┐
│ 0x0000...    │            │ 0x0000...    │
│   代码       │            │   代码       │
│   数据       │            │   数据       │
│   堆         │            │   堆         │
│   栈         │            │   栈         │
│ 0xFFF...     │            │ 0xFFF...     │
└──────────────┘            └──────────────┘
    (虚拟的)                    (虚拟的)
         ↘                    ↙
          ┌──────────────┐
          │  物理主存     │  ← 实际 DRAM 条
          │  + 磁盘交换    │
          └──────────────┘
```

> 虚拟内存在 AI Infra 里极其重要：GPU 显存管理、CUDA Unified Memory、NUMA 架构都依赖虚拟内存机制。第 9 章详讲。

### 文件

文件是对 I/O 设备的统一抽象——**一切皆文件（Everything is a file）**。

- 磁盘上的普通文件是文件
- 键盘、显示器、网络接口也是"文件"
- 程序用 `open / read / write / close` 统一接口操作所有设备

> 这就是 Unix 哲学的体现：不管连的是键盘还是网卡，程序眼里都是"一个文件"，读写就行。

## 1.8 系统之间利用网络通信

> ⚠️ 这是你之前笔记中缺失的一节。CS:APP 3e 强调：网络不仅是 I/O 设备之一，它把单台计算机的概念扩展到了分布式系统。

### 网络 = 另一类 I/O

从单机视角看，网络适配器（网卡）就是一个 I/O 设备：
- 程序通过 socket 接口读写数据
- 底层通过 DMA 在网卡和主存之间搬数据
- 和读写磁盘文件的区别：数据的另一端是**另一台机器**

### 一个扩展的 hello 程序：telnet

```
本地机器                          远程机器
┌──────────┐                    ┌──────────┐
│  telnet  │ ── 网络 ──────→   │  hello   │
│ 客户端   │ ←── 网络 ──────   │  服务端   │
└──────────┘                    └──────────┘
```

1. 用户在本地敲 `telnet remote-server`
2. telnet 客户端通过网络把输入发给远程机器
3. 远程机器上的 shell（或 telnet 服务端）收到命令，运行 hello
4. hello 的输出通过网络发回本地，显示在本地终端上

> 这就是「网络即计算机」的雏形——你在本地操作，程序在远程执行，结果传回来。云计算、分布式训练都是这个模型的放大版。

### 关键认识

- 网络让**单台计算机的边界变得模糊**
- 从 hello 的本地运行到 telnet 的远程运行，程序本身没变，变的是输入输出的路径
- 后续章节（第 11 章网络编程、第 12 章并发编程）会详细展开

> **对 AI Infra 的意义**：分布式训练 = 多台机器通过网络（InfiniBand/RoCE）同步梯度。单机内部的存储层次（1.6）和多机之间的网络通信（1.8）加在一起，构成了完整的「数据搬运全景图」。

## 1.9 重要主题

> ⚠️ 这也是你之前笔记中缺失的一节。1.9 是全章的「定调」——它把前面所有细节串成两条主线。

### 1.9.1 Amdahl 定律（Amdahl's Law）

**核心公式**：如果对系统某部分加速了 S 倍，该部分原来占总时间的比例为 α，则整体加速比为：

$$T_{new} = (1 - \alpha)T_{old} + \frac{\alpha}{S} T_{old}$$

$$Speedup = \frac{T_{old}}{T_{new}} = \frac{1}{(1 - \alpha) + \alpha / S}$$

**关键推论——边际递减效应**：
- 即使你把某部分加速到无限快（S → ∞），整体加速比也有上限：`1 / (1 - α)`
- 比如某操作占总时间的 60%（α = 0.6），你的最大加速比只有 `1 / 0.4 = 2.5×`
- **要想整体快，必须加速占时间比重大的部分**

> 现实中：GPU 把矩阵乘法做得飞快（α ≈ 0.7-0.9），所以深度学习训练整体快了。但如果数据加载占 50%（α = 0.5），GPU 再快整体也就 2×——这就是 DataLoader 优化的意义。

### 1.9.2 并发和并行

三个层次：**线程级并发 → 指令级并行 → 单指令多数据并行**

#### 线程级并发（Thread-Level Concurrency）

- **单核时代**：多个线程通过时间片轮转——假并发。进程切换有开销。
- **多核时代**：多个线程真正同时跑在不同的 CPU 核上——真并行。
- **超线程（Hyperthreading）**：一个物理核模拟两个逻辑核，共享大部分硬件但复制 PC 和寄存器。一个线程等内存时，另一个线程可以跑——提高核的利用率。

```
多核处理器示意：
┌─────┬─────┬─────┬─────┐
│Core0│Core1│Core2│Core3│
├─────┼─────┼─────┼─────┤
│ L1  │ L1  │ L1  │ L1  │
├─────┴─────┴─────┴─────┤
│    共享 L3 Cache       │
├───────────────────────┤
│      内存控制器         │
└───────────────────────┘
```

#### 指令级并行（Instruction-Level Parallelism, ILP）

- 现代 CPU 同时执行多条指令（流水线 + 超标量），每条指令从开始到结束要多个时钟周期
- **流水线（Pipelining）**：把一条指令拆成多阶段，不同指令的不同阶段重叠执行——工厂流水线
- 一个周期可以执行 2-4 条指令（超标量），前提是指令之间没有依赖关系
- 编译器帮你重排指令顺序来充分利用 ILP

#### 单指令多数据并行（SIMD）

- 一条指令同时操作多个数据——**向量化**
- 如：一条 SIMD 加法指令可以把 8 对 float 一起加（AVX 256-bit 寄存器 = 8×32bit float）
- 编译器在 `-O2` / `-O3` 下自动做 SIMD 优化（循环向量化）

> **AI Infra 的关系**：GPU 就是 SIMD 的极致——一个 warp 里的 32 个线程执行同一条指令。SIMD → SIMT（Single Instruction Multiple Thread）→ GPU kernel。

### 1.9.3 计算机系统中抽象的重要性

全书的核心方法论：**用抽象管理复杂性，每一层封装了下层细节。**

| 抽象 | 在哪章讲 | 封装了什么 |
|------|---------|-----------|
| 文件 | （本书不专讲） | I/O 设备细节 |
| 虚拟内存 | 第 9 章 | 主存 + 磁盘 I/O |
| 进程 | 第 8 章 | CPU + 主存 + I/O 设备 |
| 虚拟机 | （穿插） | 整个计算机（OS + 硬件） |
| **指令集体系结构（ISA）** | 第 3-4 章 | CPU 实际硬件实现 |

> **ISA（指令集体系结构）** 是最关键的一层抽象：它定义了 CPU 能执行哪些指令、寄存器的名字、内存寻址方式。x86-64 和 ARM 是不同的 ISA。编译器把高级语言编译到 ISA，CPU 硬件实现这个 ISA。——**软件和硬件的分界线**。

## 第一章核心总结

**两种贯穿全书的视角：**

### 视角一：一个程序的完整旅程

```
hello.c  →  预处理  →  编译  →  汇编  →  链接  →  hello (可执行文件)
                                                       ↓
                                                shell 加载到主存 (DMA)
                                                       ↓
                                                CPU 逐条取指/译码/执行
                                                ↕ (cache 减少访存延迟)
                                              寄存器 ← L1 ← L2 ← L3 ← 主存
                                                       ↓
                                               系统调用 → 写显示器
                                              （或通过网络发到远程终端）
```

### 视角二：抽象的层次塔

```
        应用程序（C / Python / ...）
              ↓ 编译器
        ISA（x86-64 / ARM）      ← 软硬件分界线
              ↓ 硬件实现
     CPU + Cache + 主存 + I/O + 网络
```

**第一章的核心画面**：计算机 = 硬件在下面跑 + 操作系统在中间管 + 程序在最上面。全书就是逐层展开这个画面——第 2 章讲数据怎么表示、第 3-4 章讲 ISA、第 5-6 章讲程序优化、第 7 章讲链接、第 8 章讲进程、第 9 章讲虚拟内存、第 10-12 章讲 I/O、网络和并发。
