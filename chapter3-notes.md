# CS:APP 第三章笔记 — 程序的机器级表示

> 基于 CS:APP 3rd Edition。第三章是整本书最核心的一章——你把 C 代码编译成汇编，然后逐条看懂 CPU 在干什么。这是「软件和硬件的分界线」落到实处的章节。

---

## 3.1 历史视角

Intel 处理器演化：**8086（1978）→ 80286 → i386（1985）→ i486 → Pentium → Core → x86-64（2003）**

只需要知道三件事：

1. **x86-64 是今天你电脑的指令集。** 你的 CPU 说的就是这种语言。
2. **向后兼容。** 1978 年 8086 上能跑的指令，今天的 x86-64 CPU 照样能跑。这是 Intel 最成功的商业策略，也是指令集越来越臃肿的原因。
3. **i386 是第一个 32 位架构**，x86-64 是 64 位扩展——寄存器从 32 位扩到 64 位，数量翻倍（8 → 16 个）。

## 3.2 程序编码

### 你写的 C → CPU 看到的机器码，经过四步

```
hello.c  →  预处理  →  编译  →  汇编  →  链接  →  可执行文件
             (cpp)     (cc1)    (as)     (ld)
```

但实际中我们只关注两条命令：

| 命令 | 做什么 | 输出 |
|------|--------|------|
| `gcc -Og -S hello.c` | 编译成汇编代码 | `hello.s`（文本文件，人类可读） |
| `gcc -Og -c hello.c` | 编译 + 汇编成机器码 | `hello.o`（二进制文件，CPU 可执行） |
| `objdump -d hello.o` | 把 `.o` 反汇编回汇编 | 机器码 + 汇编对照 |

> `-Og` = 开启基础优化但不影响调试。不要用 `-O0`（完全不优化）——生成的代码充斥无用 load/store，读起来痛苦。

---

### hello.s 文件的结构——以你亲手生成的为例

```asm
        .file   "hello.c"          ← 汇编器指令：源文件名
        .text                       ← 汇编器指令：以下是代码段

.LC0:                               ← 标签：字符串数据
        .ascii "hello\0"            ← 汇编器指令：存字符串

        .globl  main                ← 汇编器指令：main 是全局符号
        .def    main; ...
        .seh_proc main

main:                               ← 函数入口
        subq    $40, %rsp           ← CPU 指令：分配栈空间
        call    __main              ← MinGW 内部初始化
        leaq    .LC0(%rip), %rcx    ← CPU 指令：加载字符串地址到参数寄存器
        call    puts                ← CPU 指令：调用 puts 输出
        movl    $0, %eax            ← CPU 指令：return 0（%eax = 返回值）
        addq    $40, %rsp           ← CPU 指令：回收栈空间
        ret                         ← CPU 指令：返回
```

---

### 两类内容——必须一眼能区分

| 类型 | 开头 | 例子 | 谁用 |
|------|------|------|------|
| **汇编器指令** | `.` | `.file` `.text` `.globl` `.ascii` | 给工具链（as/ld）看的元信息 |
| **CPU 指令** | 操作码 | `subq` `call` `movl` `leaq` `ret` | CPU 真正执行的指令 |

---

### 你看到的关键寄存器

| 寄存器 | 全名 | 在你 hello 里的角色 |
|--------|------|-------------------|
| `%rsp` | Stack Pointer（栈指针） | 指向栈顶——函数进来先挪栈（`subq`），出去再还原（`addq`） |
| `%rcx` | 第 4 个参数寄存器 | 装 `"hello"` 的地址，传给 `puts` |
| `%eax` | 返回值（Return Value） | `return 0` → `%eax = 0` |

---

### 编译器的「自作主张」——你写 printf，它用 puts

```c
printf("hello\n");
```

编译器发现：格式字符串里没有 `%d` `%s` `%f` 等占位符，参数只有一个纯字符串。`printf` 比 `puts` 贵（要多解析格式字符串），所以帮你优化成 `puts`。

> 这是你第一次亲眼看到编译器优化——后面会遇到更多。学会读 `gcc -S` 的输出是理解 C 代码性能的第一手资料。

---

### 知识检查

能回答这三个问题，3.2 就算过了：

| 问题 | 答案 |
|------|------|
| `hello.s` 和 `hello.o` 的区别？ | `.s` 是文本汇编，`.o` 是二进制机器码 |

## 3.3 数据格式

Intel 用**指令后缀**标记操作数大小——这是你和汇编打交道时必须立刻认出的第一个特征：

| C 类型 | Intel 后缀 | 大小 | 举例 |
|--------|-----------|------|------|
| `char` | `b`（byte） | 1 字节 | `movb` `addb` |
| `short` | `w`（word） | 2 字节 | `movw` `addw` |
| `int` / `long`（32 位） | `l`（long word） | 4 字节 | `movl` `addl` |
| `long`（64 位）/ 指针 | `q`（quad word） | 8 字节 | `movq` `addq` |
| `float` | `s`（single） | 4 字节 | `movss` `addss` |
| `double` | `l`（long） | 8 字节 | `movsd` `addsd` |

> ⚠️ `l` 后缀既表示 4 字节整数又表示 8 字节浮点数——根据指令类型区分。看指令名就行，不会混淆。

**这就是你昨天 hello.s 里 `subq` `movl` `addq` 「q」「l」后缀的来源。**

---

### 寄存器命名——同一个寄存器怎么访问不同大小的部分

以 `%rax` 为例，64 位寄存器可以访问不同子集：

```
%rax ───────────────────────────────  64 位（8 字节）
              └────── %eax ────────   低 32 位（4 字节）
                      └── %ax  ───   低 16 位（2 字节）
                      └─ %ah ─       高 8 位（旧式）
                        └─ %al ─     低 8 位（1 字节）
```

**关键规则**：操作 `%eax` 时，高 32 位**自动清零**。这也是为什么 `movl $0, %eax` 等价于 `movq $0, %rax`——前者还省一个字节的指令编码。

| 你要操作 | 用哪个名字 | 举例 |
|---------|-----------|------|
| 整个 64 位（存指针） | `%rax` | `movq $0x1000, %rax` |
| 低 32 位（存 int） | `%eax` | `return 0` → `movl $0, %eax` |
| 低 16 位（基本不用） | `%ax` | — |
| 低 8 位（单个 char） | `%al` | `movb $65, %al` |

> 注：`%ah`（高 8 位）在 x86-64 中兼容性受限，新代码基本不用。

---

### x86-64 的 16 个通用寄存器

| 64 位 | 32 位 | 16 位 | 8 位 | 用途约定 |
|-------|-------|-------|------|---------|
| `%rax` | `%eax` | `%ax` | `%al` | 返回值 |
| `%rbx` | `%ebx` | `%bx` | `%bl` | 被调用者保存 |
| `%rcx` | `%ecx` | `%cx` | `%cl` | 第 4 个参数 |
| `%rdx` | `%edx` | `%dx` | `%dl` | 第 3 个参数 |
| `%rsi` | `%esi` | `%si` | `%sil` | 第 2 个参数 |
| `%rdi` | `%edi` | `%di` | `%dil` | 第 1 个参数 |
| `%rbp` | `%ebp` | `%bp` | `%bpl` | 基址指针 |
| `%rsp` | `%esp` | `%sp` | `%spl` | **栈指针** |
| `%r8`–`%r15` | `%r8d`–`%r15d` | `%r8w`–`%r15w` | `%r8b`–`%r15b` | 第 5–6 个参数 + 通用 |

> 🔑 记住 `%rsp`（栈指针）、`%rax`（返回值）、`%rdi`–`%rdx`（前 3 个参数）就够应付大多数场景。

## 3.4 访问信息

### 3.4.1 操作数指示符 —— 今天最核心的知识点

一条 `mov` 指令 = 三个部分：

```
movq  源操作数,  目的操作数
```

数据来源/去向只有三种类型：

| 类型 | 形式 | 含义 | 例子 |
|------|------|------|------|
| **立即数** | `$常数` | 就是常数本身（嵌在指令里） | `$42` `$0x1A` `$-5` |
| **寄存器** | `%reg` | 寄存器的值 | `%rax` `%rsp` `%eax` |
| **内存引用** | `偏移(基址, 变址, 比例)` | 从内存某个地址读/写 | `(%rax)` `-8(%rbp)` |

---

#### 内存寻址的完整公式

```
Imm(rb, ri, s)  →  有效地址 = Imm + R[rb] + R[ri] × s
```

| 组件 | 含义 | 是否必须 |
|------|------|---------|
| `Imm` | 常数偏移（立即数） | 否，省略 = 0 |
| `rb` | 基址寄存器（64 位） | 否，但四个至少一个 |
| `ri` | 变址寄存器（64 位） | 否 |
| `s` | 比例因子（1、2、4、8） | 否，必须和 `ri` 同时出现或省略 |

> `s` 只能是 1、2、4、8——对应 char/short/int/long 的大小。这是为数组索引设计的。

#### 常见内存寻址速查

| 形式 | 地址 = | 用在 |
|------|--------|------|
| `(%rax)` | `R[%rax]` | 解引用指针 `*ptr` |
| `8(%rsp)` | `R[%rsp] + 8` | 访问栈上第 2 个局部变量 |
| `-8(%rbp)` | `R[%rbp] - 8` | 访问栈帧中较低位置的变量 |
| `(%rax, %rcx, 4)` | `R[%rax] + R[%rcx] × 4` | 数组 `arr[i]`，`%rcx` = i，每个元素 4 字节 |
| `arr(%rip)` | `R[%rip] + arr`（相对 RIP） | 访问全局数据 / 跳转表 |

> 🔑 `8(%rsp)` 在过程调用中大量出现——每次 `call`/`ret` 都在和它打交道。3.7 节会深入。

### 3.4.2 数据传送指令

#### mov 家族

| 指令 | 效果 | 大小 |
|------|------|------|
| `movb S, D` | D ← S | 1 字节 |
| `movw S, D` | D ← S | 2 字节 |
| `movl S, D` | D ← S | 4 字节，且 D 高 32 位清零 |
| `movq S, D` | D ← S | 8 字节 |
| `movabsq I, R` | R ← I | 64 位立即数 → 寄存器 |

**核心规则**：源和目的**不能同时是内存地址**——必须至少有一个是寄存器或立即数。

```
movq (%rax), (%rbx)   ← ❌ 非法！内存→内存
movq (%rax), %rcx     ← ✅ 正确。内存→寄存器
movq %rcx, (%rbx)     ← ✅ 正确。寄存器→内存
```

> 原因：x86-64 指令集设计上不支持一条指令做两次内存访问。要搬数据，先搬到寄存器中转。

#### 零扩展传送（高位补 0）

| 指令 | 从 | 到 | 说明 |
|------|----|----|------|
| `movzbw` | 1 字节 | 2 字节 | |
| `movzbl` | 1 字节 | 4 字节 | 最常用——char → int |
| `movzwl` | 2 字节 | 4 字节 | |
| `movzbq` | 1 字节 | 8 字节 | |
| `movzwq` | 2 字节 | 8 字节 | |

> `movzbl` 把 `unsigned char` 转成 `int` 时出现——大小端无关、类型提升都靠它。

#### 符号扩展传送（高位补符号位）

| 指令 | 从 | 到 | 说明 |
|------|----|----|------|
| `movsbw` | 1 字节 | 2 字节 | |
| `movsbl` | 1 字节 | 4 字节 | |
| `movswl` | 2 字节 | 4 字节 | |
| `movsbq` | 1 字节 | 8 字节 | |
| `movswq` | 2 字节 | 8 字节 | |
| `movslq` | 4 字节 | 8 字节 | `int` → `long` 时出现 |
| `cltq` | `%eax` | `%rax` | 专用于 `%rax` 的符号扩展 |

> 🔑 `movzbl` vs `movsbl`：前者补 0（无符号），后者补符号位（有符号）。`(char)-1` 用 `movzbl` 变成 255，用 `movsbl` 变成 -1——完全不同的结果。

> ⚠️ 压栈/弹出（`pushq` / `popq`）在 §3.4.4 讲。读到那里再补笔记。

### 3.4.1-3.4.2 自检

| 问题 | 答案 |
|------|------|
| 三种操作数类型？ | 立即数（`$常数`）、寄存器（`%reg`）、内存引用（`Imm(%rb,%ri,s)`） |
| `8(%rsp, %rcx, 4)` 的地址怎么算？ | `R[%rsp] + 8 + R[%rcx] × 4` |
| `movzbl` 和 `movsbl` 的区别？ | 零扩展（高位补 0）vs 符号扩展（高位补符号位） |
| `movl $0, %eax` 为什么比 `movq $0, %rax` 更高效？ | `%eax` 操作自动清零高 32 位，且指令编码少 1 字节 |
| 为什么不能 `movq (%rax), (%rbx)`？ | x86-64 不允一条指令两次内存访问——需经过寄存器中转 |
